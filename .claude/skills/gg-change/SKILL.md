# gg-change

## 절차
1. `intake-guide`로 사용자 범위와 수정 요청을 확인한다.
2. `change-coder`가 관련 파일만 읽고 변경 범위를 분류한다.
3. 새 패키지, DB 변경, 외부통신, 개인정보, 인증 영향이 있으면 해당 산출물을 갱신한다.
4. `security-checker(quick)`로 변경 범위 중심 검사를 수행한다.
5. `_workspace/00_작업현황.md`에 변경 파일, 영향 범위, 후속조치를 남긴다.

## 원칙
- 전체 PRD/설계를 다시 만들지 않는다.
- Track 변경이 필요하면 `gg-vibecode` 신규 설계 흐름으로 승격한다.
- 패키지는 승인 카탈로그와 denylist를 따른다.
- 행정망에는 CDN/외부 API/외부 LLM을 새로 넣지 않는다.
