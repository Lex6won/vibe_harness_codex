# 공공 바이브코딩 산출물 성숙도 모델

하네스는 모든 결과물을 곧바로 정식 서비스로 보지 않는다. 현재 목표 단계를 먼저 정하고, 단계에 맞는 질문·검증·산출물 강도를 적용한다.

## 단계

| 단계 | 이름 | 목적 | 허용 범위 | 필수 산출물 |
|---|---|---|---|---|
| L0 | 아이디어 구체화 | 업무 문제를 화면·기능·데이터로 정리 | 코드 없음 또는 설계만 | feature brief, PRD 초안 |
| L1 | 시제품 | 내부 데모, 업무 가능성 확인 | 더미 데이터, 제한 사용, 외부 공개 금지 | PRD, 화면/기능, 개발스택 초안, 소스 |
| L2 | 내부도구 | 부서/기관 내 실제 업무 보조 | 행정망 기준, 승인 패키지, 기초 보안점검 | PRD, DB, 개발스택 최종, quick/standard 검증 |
| L3 | 정식 서비스 후보 | 운영팀/보안팀 이관 준비 | 배포신청 전 단계, full MCP 검증 필요 | MCP 결과, 배포가이드, 신청서 초안 |
| L4 | 정식 운영 | 승인된 환경에서 운영 | 사람 승인, 운영절차, 보안성검토 완료 필요 | 승인 문서, 운영계획, 배포 패키지 |

## 원칙

- L1을 L4처럼 취급하지 않는다. 절차 과부하가 바이브코딩 활성화를 막는다.
- L1/L2를 L4로 착각하지 않는다. 실제 개인정보·시민 공개·기관 공식 서비스는 별도 승인이 필요하다.
- 사용자가 목표 단계를 모르면 기본값은 L1 시제품이다.
- 개인정보, 시민 접근, 외부통신, 파일업로드, DB 저장이 있으면 최소 L2 이상으로 올리고 검증을 강화한다.
- 배포·이관 요청이 있으면 L3 흐름으로 승격한다.

## 사용자에게 보여줄 표현

- “현재는 내부 시제품 단계입니다.”
- “실제 개인정보를 넣거나 시민에게 공개하려면 배포·이관 준비 단계가 필요합니다.”
- “지금은 정식 승인 완료가 아니라 제출 준비 완료 상태입니다.”
